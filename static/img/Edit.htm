Registration Service

Learn more

In order to send push notifications to specific users, or groups of users, you will need a way to associate users with their devices, or more specifically, the push tokens used to send push notifications to their devices.

To accomplish this, you may configure a "registration service" that will POST the user's push token to an API endpoint on your server. The POST request will come from the user, so you'll be able to identify which user the push token connects to.

Specify in regular expression. Data will also be POSTED when push tokens are initially created.

We strongly recommend specifying .* for testing, at least until you have verified you can receive a POST. If you do not receive the POST, please verify your endpoint can receive a POST request by testing with a tool like Curl or Postman.

Finally, make sure you log the entire POST request, including headers. Any session cookie that identifies the user will come across in the headers of the request. If you are using PHP, you can use something like: $data = file_get_contents('php://input'); $data = json_decode($data);

Deep Linking

Learn more

When deep linking is enabled, and a user clicks a link on their device matching a domain specified below, the user will be prompted to open the link within your app rather than in the mobile browser.

For Android, this will work automatically with no further configuration.

For Apple, you will need to add a configuration file to your website at /apple-app-site-association to prove you control the referenced domain. Learn more at Apple Documentation.

Most likely, you will want to add your hostname with and without the 'www' prefix. For example, to support Deep Links on GoNative.io, you would add 'gonative.io', and 'www.gonative.io' below.

Webview Settings

Support for window.open()

Learn more

When enabled, your app will handle window.open() javascript similar to a browser by opening a new tab. This is required to support certain social login flows. When disabled, the URLs specified in window.open() will load in the current Webview as if the user clicked on a regular link.



Screen Orientation

By default, your app will rotate portrait->landscape or landscape->portrait when the device rotates. You may optionally specify portrait-only or landscape-only.



iOS Webview Settings

Learn more

For iOS, we use WKWebView by default for iOS 8.x and iOS 9.x, and use UIWebView for iOS 7.x. You may optionally force UIWebView for all iOS versions. The main reason for this is that the newer WKWebView does not support HTML5 Cache Manifest.



Android Webview Settings

Learn more

For Android, we use standard Android Webview by default, and offer Crosswalk Webview as a premium feature. If enabled, your license must include Crosswalk Webview for Android. More info at pricing.



Android requires enabling a permission in your app's manifest to access user's geolocation.

Pre-load Website URLs (Advanced)

"Webview Pools" allow your app to pre-load specified URLs for faster performance. When any URL in a pool is loaded, your app will automatically prepare the other URLs in the pool for use.

NOTE: Webview Pools will not work well with Single Page Apps.

For example, specify a pool that includes your Initial URL so that all pages will be pre-loaded in the background when your app first launches.

For each URL, you may specify when it should be refreshed.
